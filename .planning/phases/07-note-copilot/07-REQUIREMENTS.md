# Phase 7 Requirements: Note Copilot

**Phase:** 07-note-copilot
**Requirements:** F11-01 through F11-08

---

## F11-01 — Copilot Panel (per-note)

**Requirement:** User can open a copilot panel on any note.

**Details:**
- Side panel (right of NoteView) toggleable via toolbar button
- State managed in `uiStore.ts` (`copilotPanelOpen: boolean`)
- Panel has 3 tabs: Chat, Suggestions, Proposal
- Persists open/closed state during session

**Acceptance Criteria:**
- [ ] Toolbar button (sparkle icon) visible in NoteView header
- [ ] Clicking button toggles copilot side panel
- [ ] Panel slides in/out smoothly (CSS transition)
- [ ] Panel shows correct tab content per active tab

---

## F11-02 — Explain Action

**Requirement:** "Explain" returns accurate note summary in context.

**Details:**
- Uses `RetrievalService.build_context_pack()` to get note chunk + neighbors
- PydanticAI agent generates explanation with reference to specific sections
- Output is Markdown, references heading path of relevant chunks

**Acceptance Criteria:**
- [ ] `POST /copilot/explain/{note_id}` returns Markdown explanation
- [ ] Explanation references specific sections/headings from the note
- [ ] Context pack provides surrounding chunks for cross-references
- [ ] Response time < 5s for typical note

---

## F11-03 — Summarize Action

**Requirement:** "Summarize" produces concise recap of the note.

**Details:**
- Takes note content + optional context pack
- PydanticAI generates 2-3 paragraph summary
- Highlights key decisions, open questions, next steps if identifiable

**Acceptance Criteria:**
- [ ] `POST /copilot/summarize/{note_id}` returns Markdown summary
- [ ] Summary is 100-300 words
- [ ] If note has frontmatter (tags, dates), those are incorporated
- [ ] Summary is idempotent (same note → same summary, stable model)

---

## F11-04 — Suggest Links Action

**Requirement:** "Suggest links" proposes relevant internal links with reasoning.

**Details:**
- Uses `RetrievalService.hybrid_search()` to find related notes
- For each suggested link: `{ target_note_path, target_title, reason }`
- Returns top 5 suggestions ordered by relevance
- Each suggestion includes why it relates to the current note

**Acceptance Criteria:**
- [ ] `POST /copilot/suggest-links/{note_id}` returns list of suggestions
- [ ] Each suggestion includes: note_path, title, reason (1-2 sentences)
- [ ] Uses hybrid search to find semantically related notes
- [ ] Excludes already-linked notes (checks existing wiki-links in content)

---

## F11-05 — Suggest Tags Action

**Requirement:** "Suggest tags" proposes appropriate tags for the note.

**Details:**
- Analyzes note content (title + headings + body)
- Cross-references with existing workspace tags via RetrievalService
- Returns ranked list of suggested tags: `[{ tag, confidence: 0.0-1.0, reason }]`

**Acceptance Criteria:**
- [ ] `POST /copilot/suggest-tags/{note_id}` returns tag suggestions
- [ ] Each suggestion includes: tag, confidence score, reason
- [ ] Existing tags from frontmatter are excluded
- [ ] Top 5 suggestions returned

---

## F11-06 — Suggest Structure Action

**Requirement:** "Suggest structure" identifies note structure improvement opportunities.

**Details:**
- Analyzes note heading tree and content flow
- Identifies issues: missing headings, deeply nested sections, long paragraphs, logical gaps
- Returns structured list: `[{ type, location, issue, suggestion }]`

**Types:**
- `missing_heading` — logical section has no heading
- `deeply_nested` — H1 > 3 levels without break
- `long_paragraph` — paragraph > 200 words could be a list/heading
- `logical_gap` — abrupt topic change without heading

**Acceptance Criteria:**
- [ ] `POST /copilot/suggest-structure/{note_id}` returns structured suggestions
- [ ] Each suggestion: `{ type, location, issue, suggestion }`
- [ ] Covers all 4 issue types
- [ ] Location references heading path or line range

---

## F11-07 — Propose Patch Action

**Requirement:** "Propose patch" generates a diff/patch instead of direct edit.

**Details:**
- User enters change instruction (e.g., "Add a section about deployment")
- PydanticAI generates a unified diff against current note content
- Diff is submitted to `ProposalService.create_proposal()` → Exchange Zone
- Returns proposal ID so frontend can redirect to Exchange Zone

**Security:** The patch is generated by AI, not applied directly. User must approve in Exchange Zone.

**Acceptance Criteria:**
- [ ] `POST /copilot/propose-patch/{note_id}` accepts `{ instruction: str }`
- [ ] Returns `{ proposal_id: UUID, diff: str }` (not applied)
- [ ] Diff is valid unified diff format
- [ ] Proposal appears in Exchange Zone for user approval

---

## F11-08 — UX Mode Distinction

**Requirement:** UX clearly distinguishes: conversation (chat), suggestion (recommendation), proposal (patch-to-approve).

**Details:**
- Three distinct visual modes in the copilot panel:
  1. **Chat** — chat bubble UI, input field at bottom, stateless
  2. **Suggestion** — card/button grid, each action (Explain, Summarize, etc.) as a button
  3. **Proposal** — diff viewer UI showing generated patch
- Visual design must make the distinction immediately obvious
- Chat should feel transient (no persistence)
- Suggestion should feel advisory (AI recommends, user decides)
- Proposal should feel formal (this WILL change your note if approved)

**Acceptance Criteria:**
- [ ] Tab navigation between Chat/Suggestions/Proposal
- [ ] Chat: input field, send button, chat bubble list
- [ ] Suggestions: action button grid (Explain, Summarize, Suggest Links, etc.)
- [ ] Proposal: diff viewer with before/after panes
- [ ] Proposal tab shows clear "Submit to Exchange Zone" CTA
- [ ] No mode conflation (chat history doesn't mix with suggestion cards)

## New Requirements Discovered During Planning

### F11-09 — Session Isolation

Chat mode is stateless — each message is independent. No conversation memory between turns. Rationale: the agent has no persistent session context in the copilot; it only knows the current note.

### F11-10 — Error Handling

If LLM call fails, return user-friendly error in the panel (not a generic 500). Show: "Couldn't generate explanation. Try again." with a retry button.

### F11-11 — Loading States

Each action shows a loading indicator (spinner/skeleton) while awaiting LLM response. No timeout < 30s (notes can be long).
