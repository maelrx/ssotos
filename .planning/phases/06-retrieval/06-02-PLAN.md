## Task 5: Chunking Service

**Files created:** `src/services/chunking_service.py`

### Context
- No chunking service exists yet
- Per D-101: heading-guided splitting on H1, H2, H3 boundaries
- Per D-102: max chunk size 512 tokens, overlap 50 tokens
- Per D-103: overlap for context continuity

### Action
Create `src/services/chunking_service.py` implementing:

1. **`ChunkResult` dataclass** with: `content`, `heading_path`, `chunk_index`, `char_start`, `char_end`, `token_count`

2. **`count_tokens(text, model)`** — use tiktoken for accurate counting (or fallback to approximation for non-OpenAI models)

3. **`find_break_point(text, start, end)`** — find good split point using priority:
   - Paragraph break (`\n\n`) first
   - Sentence break (`. `, `! `, `? `) second
   - Word boundary last
   - Return the break position or `end` if none found

4. **`chunk_by_headings(markdown, max_tokens=512, overlap_tokens=50)`**:
   - Split markdown at H1 (`#`), H2 (`##`), H3 (`###`) boundaries
   - Build heading path hierarchy
   - For each section: if tokens <= max_tokens, create one chunk
   - If section > max_tokens: sub-split with overlap, using `find_break_point`
   - Merge very short sections (<100 tokens) with next section
   - Track `char_start` and `char_end` for provenance
   - Return list of `ChunkResult`

5. **`markdown_to_chunks(markdown, note_projection_id)`** — convert markdown to full Chunk DB records:
   - Run `chunk_by_headings` on the markdown body
   - Parse and strip YAML frontmatter from input (handles `---...---` delimiters)
   - Return list of `Chunk` model instances ready for insertion

### Acceptance Criteria
- [ ] `src/services/chunking_service.py` created
- [ ] `chunk_by_headings` splits correctly at H1/H2/H3 boundaries
- [ ] `max_tokens=512` enforced (configurable per note kind but default 512)
- [ ] `overlap_tokens=50` maintained between consecutive chunks
- [ ] `find_break_point` prefers paragraph > sentence > word boundary
- [ ] `char_start` and `char_end` correctly track position in original markdown
- [ ] Sections <100 tokens are merged with next section
- [ ] Frontmatter (`---...---`) is stripped before chunking
- [ ] Function returns `list[ChunkResult]` with all fields populated

## Task 6: Embedding Service

**Files created:** `src/services/embedding_service.py`

### Context
- `EmbeddingsProvider` protocol already defined in `retrieval_service.py` (Wave 1)
- Need batch embedding generation with concurrency control per D-95, D-96

### Action
Create `src/services/embedding_service.py`:

1. **`EmbeddingResult` dataclass** — `(chunk_id: UUID, embedding_vector: list[float], model: str)`

2. **`generate_embeddings_batch(provider, chunks, batch_size=100, max_concurrent=5)`**:
   - Takes list of `(chunk_id, content_text)` pairs
   - Processes in batches of 100 (per D-96)
   - Uses `asyncio.Semaphore(max_concurrent)` to limit concurrent API calls to 5
   - Calls `provider.embed_texts(texts)` for each batch
   - Returns `list[EmbeddingResult]`

3. **Global `get_embeddings_provider()` factory**:
   - Reads `OPENAI_API_KEY` from environment
   - Returns `OpenAIEmbeddingsProvider` instance
   - Caches in module-level variable

4. **Note: OpenAI API call** uses httpx async client directly (not the full openai SDK — lighter per Stack Decision Record)

### Acceptance Criteria
- [ ] `src/services/embedding_service.py` created
- [ ] `generate_embeddings_batch` processes in batches of 100
- [ ] Concurrency limited to 5 simultaneous API calls via semaphore
- [ ] `get_embeddings_provider()` returns cached `OpenAIEmbeddingsProvider`
- [ ] Uses `text-embedding-3-small` model, 1536 dimensions
- [ ] Returns `list[EmbeddingResult]` with chunk_id, vector, model
- [ ] API errors raise descriptive exceptions (not silently swallowed)

## Task 7: Hybrid Search Integration

**Files modified:** `src/services/retrieval_service.py` (augment Wave 1 implementation)

### Context
- Wave 1 created the `RetrievalService` skeleton
- Wave 2 connects chunking and embedding into the hybrid search flow

### Action
Add to `src/services/retrieval_service.py`:

1. **`search(query, workspace_id, limit=20, mode="hybrid")`** — main entry point:
   - If mode is "fts" or "hybrid": get FTS results
   - If mode is "vector" or "hybrid": get query embedding, then get vector results
   - If mode is "hybrid": apply RRF to combine
   - If only one mode: return single-ranker results with score breakdown
   - Build `SearchResult` list with all fields

2. **`get_fts_results(query, workspace_id, limit)`** — wrapper calling `fts_search`

3. **`get_vector_results(query_vector, workspace_id, limit)`** — wrapper calling `vector_search`

4. **`build_score_breakdown(fts_score, vector_score, note_updated_at)`** — per D-108:
   - `fts_component`: fts_score * 0.5
   - `vector_component`: vector_score * 0.5
   - `fts_raw`, `vector_raw`
   - `recency_boost`: +0.05 if note updated within 7 days

5. **`generate_why_matched(query, chunk, fts_score, vector_score)`** — human-readable explanation:
   - If FTS rank is high: mention keyword matches
   - If vector rank is high: mention semantic similarity
   - Format: "Matched on [keywords] (FTS rank N) with semantic similarity (vector rank N)"

### Acceptance Criteria
- [ ] `search()` method accepts `query`, `workspace_id`, `limit`, `mode`
- [ ] Mode "fts" returns only FTS results with scores
- [ ] Mode "vector" returns only vector results with scores
- [ ] Mode "hybrid" combines via RRF with k=60
- [ ] Score breakdown includes fts_component, vector_component, recency_boost
- [ ] `why_matched` field is a human-readable string explaining match reasons
